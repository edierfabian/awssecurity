    Categoría: SEC – Protección de Datos

    Como parte de las políticas de seguridad de su nueva cuenta de AWS, habilitó el cifrado predeterminado para los volúmenes EBS, de modo que todas las nuevas instancias de EC2 tengan sus volúmenes EBS cifrados automáticamente. Creó una clave KMS simétrica para cifrar todos los volúmenes. Tras unos días de pruebas, creó sus instancias de EC2, pero borró accidentalmente la clave KMS utilizada para cifrar sus volúmenes EBS.

    ¿Cuál de las siguientes acciones le ayudará a recuperar datos de todos sus volúmenes cifrados?

    • 1.
    • 2.
    • 3.
    • 4.

    Correcto

    Incorrecto

    Cuando crea un recurso EBS cifrado, este se cifra con la clave predeterminada de su cuenta para el cifrado EBS, a menos que especifique una clave KMS administrada por el cliente diferente en los parámetros de creación de volumen o en la asignación de dispositivos de bloque para la AMI o la instancia.

    Suponiendo que aún tiene un EC2 en ejecución con acceso al volumen cifrado y que tiene conectado un volumen sin cifrar, migre los datos de ese volumen cifrado al volumen sin cifrar. Puede transferir datos libremente entre ellos. EC2 realiza las operaciones de cifrado y descifrado de forma transparente.

    Por ejemplo, use el comando rsync para copiar los datos. En el siguiente comando, los datos de origen se encuentran en /mnt/source y el volumen de destino se monta en /mnt/destination.

    [ec2-user ~]$ sudo rsync -avh --progress /mnt/source/ /mnt/destination/

    De esta manera, incluso si pierde la clave KMS utilizada para cifrar el volumen EBS original, aún puede recuperar los datos y copiarlos a otro volumen sin cifrarlos.

    Por lo tanto, la respuesta correcta es:  Migrar los datos del volumen cifrado a un volumen no cifrado.

    La opción " Regenerar su clave KMS y actualizar la clave predeterminada" en la consola de AWS EC2 es incorrecta, ya que no se puede cambiar la clave KMS asociada a una instantánea o un volumen cifrado existente. Sin embargo, sí se puede asociar una clave KMS diferente durante la copia de una instantánea para que la instantánea resultante se cifre con la nueva clave KMS.

    La opción " Deshabilitar el cifrado de su volumen específico" en la consola de AWS EC2 es incorrecta, ya que no se puede eliminar el cifrado de un volumen o una instantánea cifrados. Tenga en cuenta que un volumen restaurado a partir de una instantánea cifrada, o una copia de una instantánea cifrada, siempre está cifrado.

    La opción que dice: Contactar con el soporte de AWS para regenerar la clave KMS es incorrecta porque AWS no puede recuperar ni regenerar su clave anterior. Una vez eliminada, no se puede recuperar.

    https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/AmazonEBS.html

    ¿Qué protocolo de correo electrónico y número de puerto deben incluirse en el punto final de SES? (Seleccione DOS).

    • 1.
    • 2.
    • 3.
    • 4.
    • 5.

    Correcto

    Incorrecto

    El punto de enlace SMTP de Amazon SES requiere que todas las conexiones estén cifradas mediante Seguridad de la Capa de Transporte (TLS). (Tenga en cuenta que TLS suele denominarse por el nombre de su protocolo predecesor, SSL). Amazon SES admite dos mecanismos para establecer una conexión cifrada con TLS: STARTTLS y TLS Wrapper.

    STARTTLS permite actualizar una conexión no cifrada a una cifrada. Para configurar una conexión STARTTLS, el cliente SMTP se conecta al punto de enlace SMTP de Amazon SES en el puerto 25, 587 o 2587 , emite un comando EHLO y espera a que el servidor anuncie que admite la extensión SMTP STARTTLS. A continuación, el cliente emite el comando STARTTLS, iniciando la negociación TLS. Una vez finalizada la negociación, el cliente emite un comando EHLO a través de la nueva conexión cifrada y la sesión SMTP continúa con normalidad.

    Por lo tanto, las respuestas correctas son:

    – Protocolo simple de transferencia de correo (SMTP)

    – Número de puerto 587

    El protocolo de oficina postal versión 3 (POP3)  es incorrecto. Este protocolo se utiliza para recibir correos electrónicos, no para enviarlos. Además, Amazon SES no cuenta con un servidor POP.

    El número de puerto 995 es incorrecto ya que este número de puerto está reservado para conexiones POP3 cifradas.

    El puerto 143  es incorrecto. Este puerto se utiliza para conexiones IMAP (Protocolo de acceso a mensajes de Internet).

    3 . Pregunta
    Categoría: SEC – Protección de Datos

    Un ingeniero de seguridad almacena información confidencial de cada usuario en una tabla de Amazon DynamoDB. Las entradas se cifran mediante una clave administrada por el cliente para garantizar que otros usuarios no puedan leer la información confidencial. Sin embargo, otros usuarios con privilegios podrían tener permisos de lectura y escritura en la tabla de DynamoDB. Estos usuarios podrían intercambiar entradas cifradas en la tabla y verlas. Para evitarlo, el ingeniero debe habilitar el cifrado autenticado incluyendo datos autenticados adicionales (AAD), como la dirección de correo electrónico y el nombre de usuario, en el texto cifrado.

    ¿Cuál de las siguientes opciones ayudará a lograr esto?

    • 1.
    • 2.
    • 3.
    • 4.

    Correcto

    Incorrecto

    Todas las operaciones criptográficas de AWS KMS con claves de cifrado simétrico aceptan un contexto de cifrado, un conjunto opcional de pares clave-valor que puede contener información contextual adicional sobre los datos. AWS KMS utiliza el contexto de cifrado como datos autenticados adicionales (AAD) para facilitar el cifrado autenticado.

    No se puede especificar un contexto de cifrado en una operación criptográfica con una clave KMS asimétrica. Los algoritmos de cifrado asimétrico estándar que utiliza AWS KMS no admiten un contexto de cifrado.

    Al incluir un contexto de cifrado en una solicitud de cifrado, este se vincula criptográficamente al texto cifrado, de modo que se requiere el mismo contexto para descifrar (o descifrar y volver a cifrar) los datos. Si el contexto de cifrado proporcionado en la solicitud no coincide exactamente y distingue entre mayúsculas y minúsculas, la solicitud falla. Solo puede variar el orden de los pares clave-valor en el contexto de cifrado.

    El contexto de cifrado no es secreto. Aparece en texto plano en los registros de AWS CloudTrail, lo que permite identificar y categorizar las operaciones criptográficas.

    Un contexto de cifrado puede constar de cualquier clave y valor. Sin embargo, dado que no es secreto ni está cifrado, no debe incluir información confidencial. Se recomienda que describa los datos que se cifran o descifran. Por ejemplo, al cifrar un archivo, podría usar parte de la ruta del archivo como contexto de cifrado.

    Por lo tanto, la respuesta correcta es: Úselo EncryptionContextpara solicitar información adicional, como la dirección de correo electrónico y el nombre de usuario, al descifrar sus datos.

    The option that says: Use EnvelopeEncryption for an additional layer of protection to your encrypted data. This will require the email address and username entry to correctly decrypt your data is incorrect. When you encrypt your data, they are already protected, but you also have to protect your encryption key. Envelope encryption is just the practice of encrypting plaintext data with a data key and then encrypting the data key under another key. Envelop encryption does not add additional encryption to your data. It merely protects (encrypts) the data key that you used to encrypt your data.

    The option that says: Use AuthenticationContext to require context information such as the email address and username when decrypting your data is incorrect because AuthenticationContext is only applicable in Microsoft Entra ID and not in AWS KMS.

    The option that says: Use a Master Key for an additional layer of encryption for your encrypted data. This will require the email address and username to correctly decrypt your data is incorrect. The Master Key is the one used to encrypt the data key in AWS KMS. It does not directly protect the decryption of the encrypted data itself. It doesn’t use additional authenticated data (AAD) to support authenticated encryption.

    https://docs.aws.amazon.com/kms/latest/developerguide/concepts.html#encrypt_context

    A company plans to use AWS Key Management Service (KMS) to encrypt confidential documents owned by different organizations. The Security team must have full control over how the KMS keys are used, however, they don’t want the operational overhead of rotating keys annually.

    Which AWS KMS key type should the team use to satisfy this requirement?

    • 1.
    • 2.
    • 3.
    • 4.

    Correct

    Incorrect

    Cryptographic best practices discourage extensive reuse of encryption keys. To create new cryptographic material for your AWS Key Management Service (AWS KMS) KMS keys, you can create a new KMS key and then change your applications or aliases to use the new KMS key. Or, you can enable automatic key rotation for an existing customer-managed CMK.

    When you enable automatic key rotation for a customer-managed key, AWS KMS generates new cryptographic material for the key every year. AWS KMS also saves the KMS key’s older cryptographic material in perpetuity so it can be used to decrypt data that it encrypted. AWS KMS does not delete any rotated key material until you delete the CMK. Key rotation changes only the CMK’s backing key, which is the cryptographic material that is used in encryption operations. The CMK is the same logical resource, regardless of whether or how many times its backing key changes.

If the backing key is less than 365 days old, AWS KMS resumes the original key rotation schedule.

    -KMS keys pending deletion. While a KMS key is pending deletion, AWS KMS does not rotate it. The key rotation status is set to false, and you cannot change it while deletion is pending. If deletion is canceled, the previous key rotation status is restored. If the backing key is more than 365 days old, AWS KMS rotates it immediately and every 365 days thereafter. If the backing key is less than 365 days old, AWS KMS resumes the original key rotation schedule.

    -AWS managed KMS keys. AWS KMS automatically rotates AWS managed keys every year (approximately 365 days). You cannot enable or disable key rotation for AWS managed keys.

    -AWS owned KMS keys. You cannot manage key rotation for AWS owned KMS keys. The key rotation strategy for an AWS owned CMK is determined by the AWS service that creates and manages the KMS key.

    -Monitoring key rotation. When AWS KMS automatically rotates the key material for an AWS managed key or customer-managed key, it writes the KMS key Rotation event to Amazon CloudWatch Events. You can use this event to verify that the key was rotated.

    -Unsupported KMS key types. Automatic key rotation is not supported on the following types of KMS keys, but you can rotate these keys manually.

    1. Asymmetric KMS keys
    2. KMS keys in custom key stores
    3. KMS keys that have imported key material
